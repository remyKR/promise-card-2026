## 2026-04-23

### 도구 선택
- **결정**: Claude Design 대신 **Claude Code**로 개발 진행
- **이유**: 직접 Figma 디자인 가능 + 풀스택(프론트+백엔드) 모두 필요하기 때문

### 개발 방식
- **결정**: **바이브 코딩** 방식 — Figma 디자인 후 Claude Code로 구현
- **팀 구성**: 본인(디자인·기획) + 원희님(프론트) + Claude(구현 보조)

### 레포지토리 전환
- **결정**: 구 레포(`remyKR/promisecard`)를 현재 레포(`remyKR/promise-card-2026`)로 마이그레이션
- **이유**: 기존 기획 문서는 활용하되 디자인 시스템 파일은 초기화하고 재시작

### 기술 스택 확정
- **결정**: Next.js 15 + React 19 + Tailwind CSS 4 + Supabase + pnpm
- **제외 확정**: `next-intl` (다국어 불필요, 베트남어 단일 언어로 진행)
- **이유**: 단일 언어(베트남어)로 충분, 복잡도 줄이기

### 비즈니스 방향
- **결정**: **베트남 법인** 설립 예정 (거주 비자 목적 포함)
- **결정**: 초기 MVP는 **무료 서비스**로 출시 → 법인 설립 후 결제 연동(Phase 2)
- **타겟 시장**: 베트남 거주 커플 (한국인 + 베트남인)

### 레퍼런스
- **결정**: 데어무드(`theirmood.com`) 참고하되 일부 고난이도 기능 제외
- **제외 기능(MVP)**: Zalo 로그인, 결제 연동, 배경음악, 오프닝 애니메이션, 하객 사진 QR

### 개발 일정 (예상)
- **결정**: MVP 기준 6주 목표
  - 1주차: 환경 세팅, Supabase 연결, 로그인
  - 2주차: 청첩장 에디터 폼
  - 3주차: 템플릿 3종, 실시간 미리보기
  - 4주차: 갤러리, 지도, 공유 URL
  - 5주차: RSVP, 방명록, 마이페이지
  - 6주차: Vercel 배포, 버그 수정

### 문서 구조 확정

- **결정**: "기획·정책·설계 우선" — 문서 완성 후 구현 시작
- **이유**: 설계 없는 바이브 코딩은 기술 부채와 재작업을 유발함
- **적용**: README.md 상단에 명시, CLAUDE.md에 규칙 추가

### 문서 구조 재편
- **결정**: docs 폴더를 `01-기획 / 02-정책 / 03-설계` 3계층으로 재구성
- **이유**: 기획·정책·설계의 역할 분리로 문서 찾기·관리 효율화
- **변경 내용**:
  - `01-기획/`: 서비스-개요.md, MVP-범위.md, 로드맵.md (신규)
  - `02-정책/`: 인증-정책.md, 서비스-정책.md, 운영-정책.md (신규)

  - 삭제: 01-plan/, 02-design/, 05-tech_doc/ (구 폴더)

### 타겟 사용자 명확화
- **결정**: 서비스 핵심 타겟은 **베트남인 커플 (베트남인 × 베트남인)**
- **이유**: 초기 문서에 "베트남 거주 한국인 커플 1순위"로 기재되어 있었으나, 실제 시장 볼륨은 베트남인 커플이 훨씬 큼
- **제외**: 한국인 × 베트남인 혼합 커플은 타겟 아님

### 청첩장 복사하기 기능 추가 (P1)
- **결정**: "청첩장 복사하기" 기능을 MVP P1으로 추가
- **이유**: 베트남 결혼 문화상 60~70%의 커플이 신부 고향·신랑 고향 각각 결혼식을 2회 진행. 두 청첩장은 장소·날짜만 다르고 나머지(사진·테마·계좌번호)는 동일 → 복사 기능 없으면 처음부터 다시 만들어야 하는 불편함 발생
- **비즈니스 임팩트**: 건당 결제 모델 기준 커플당 평균 1.6~2배 매출 발생
