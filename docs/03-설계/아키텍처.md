# 시스템 아키텍처

> 작성일: 2026-04-23 | 버전: v3.0  
> 이 문서는 시스템 전체 구조와 각 영역 간 관계를 다룬다.  
> 세부 규칙은 각 전담 문서를 참조한다.

---

## 1. 시스템 구조

```
[사용자 (모바일 브라우저)]
         ↓
[Vercel Edge Network]
         ↓
[Next.js 15 App Router]
  ├── Server Components    (기본 — 서버에서 렌더링)
  ├── Client Components    ('use client' 명시 시만)
  └── Server Actions       (DB 조작, 파일 업로드)
         ↓
     [Supabase]
      ├── Auth      ─── Google / Facebook OAuth + 전화번호 OTP (Twilio Verify)
      ├── Database  ─── PostgreSQL + RLS
      └── Storage   ─── 청첩장 이미지
```

---

## 2. 인증 흐름

```
소셜 로그인 (Google / Facebook)  ← Zalo는 v2
        ↓
전화번호 입력 (+84 베트남)
        ↓
SMS OTP 인증 (Decree 147/2024 준수)
        ↓
서비스 진입
```

> 상세 정책: `02-정책/인증-정책.md`

## 4. 주요 URL 구조

| URL | 설명 | 인증 필요 |
|---|---|---|
| `/` | 랜딩 페이지 | ❌ |
| `/login` | 로그인 | ❌ |
| `/verify-phone` | 전화번호 OTP 인증 | 부분 (소셜 인증 후) |
| `/create` | 청첩장 에디터 | ✅ |
| `/my` | 마이페이지 | ✅ |
| `/my/cards` | 내 청첩장 목록 | ✅ |
| `/card/[code]` | 청첩장 공개 열람 | ❌ |

> 화면별 기능 명세: `03-설계/화면명세/`

## 5. API 엔드포인트

| Method | Endpoint | 인증 | 설명 |
|---|---|---|---|
| GET | `/api/cards` | ✅ | 내 청첩장 목록 |
| POST | `/api/cards` | ✅ | 청첩장 생성 |
| GET | `/api/cards/[id]` | ✅ | 청첩장 조회 |
| PUT | `/api/cards/[id]` | ✅ | 청첩장 수정 |
| DELETE | `/api/cards/[id]` | ✅ | 청첩장 삭제 |
| GET | `/api/cards/code/[code]` | ❌ | 공개 열람 |
| POST | `/api/upload` | ✅ | 이미지 업로드 |
| POST | `/api/rsvp` | ❌ | RSVP 제출 |
| GET | `/api/rsvp/[cardId]` | ✅ | RSVP 집계 |
| POST | `/api/guestbook` | ❌ | 방명록 작성 |
| GET | `/api/guestbook/[cardId]` | ❌ | 방명록 목록 |

## 6. 보안 원칙

| 항목 | 방식 |
|---|---|
| DB 접근 제어 | Supabase RLS — 본인 데이터만 수정·삭제 |
| 이미지 접근 | Supabase Storage 버킷 정책 |
| 환경변수 | `.env.local` (git 제외) |
| 세션 | Supabase JWT + Refresh Token 자동 갱신 (7일) |
| SMS OTP | 서버 사이드 전용 처리 |
